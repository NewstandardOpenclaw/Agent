Topic 1: API Gateway Lambda オーソライザー（JWT / Lambda Authorizer）の設計

定義
  • JWT オーソライザーは、Cognito や外部 IdP が発行する JWT の署名・クレームを API Gateway 側で検証する機能。
  • Lambda Authorizerは、任意ロジックで認可判定し IAM ポリシーを返す拡張型オーソライザー。
重要性
  • DVA では「どの認証方式が運用コストと拡張性を満たすか」を問う設計問題が頻出。
  • 認可を API Gateway 入口で止めることで、Lambda 本体の処理コストとリスクを削減できる。
使い分け
  • 標準的な JWT 検証で足りるなら JWT オーソライザー（運用軽量）。
  • ヘッダー・パス・外部DB参照を含む複雑な判定なら Lambda Authorizer。
  • 高トラフィック時は Authorizer キャッシュ戦略（TTL）を設計してレイテンシを抑える。
よくある誤解
  • 「Lambda Authorizer は必ず遅い」→ キャッシュを有効化し判定を軽量化すれば実用的。
  • 「JWT 検証だけで認可が完結する」→ スコープやクレームの業務ロジック評価は別途必要な場合がある。
3分要点
  1. まず JWT で足りるかを確認し、要件過多なら Lambda Authorizer。
  2. 複雑な認可ほどキャッシュ設計が得点ポイント。
  3. 認証（本人確認）と認可（権限判定）を混同しない。

Topic 2: EventBridge リトライ / DLQ と Lambda 非同期再試行の責務分離

定義
  • EventBridge はルールベースでイベント配送し、失敗時の再試行や DLQ 設定が可能。
  • Lambda 非同期呼び出し も独自の再試行と失敗先（DLQ / On-failure destination）を持つ。
重要性
  • DVA では「どこで再試行し、どこで最終失敗を受けるか」の責務整理が問われる。
  • 責務が曖昧だと、重複実行・障害解析の複雑化・運用コスト増につながる。
使い分け
  • イベントルーティング共通基盤として失敗管理したいなら EventBridge 側 DLQ を中心に設計。
  • 関数ごとに細かい再実行制御が必要なら Lambda 側非同期設定 を調整。
  • 両方使う場合は「一次失敗の責任境界」を明文化し、監視メトリクスを分離する。
よくある誤解
  • 「DLQを二重に設定すれば安全性が上がる」→ 追跡経路が複雑化し、復旧手順が逆に不明瞭になる。
  • 「再試行は多いほどよい」→ 下流障害時にスロットリング連鎖を起こすため、上限とバックオフが重要。
3分要点
  1. 再試行責務は EventBridge か Lambda かを先に決める。
  2. DLQ は“最終退避先”を1本化するほうが運用しやすい。
  3. 監視は失敗段階（配送失敗/実行失敗）で分ける。
