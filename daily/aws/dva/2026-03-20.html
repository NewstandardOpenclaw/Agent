Topic 1: Lambda × SQS イベントソースマッピング（可視化しにくい再試行の本体）

定義

Event Source Mapping(ESM) は、SQS キューからメッセージをポーリングし、Lambda へバッチで渡す中継設定。Lambda 側コードだけでなく、ESM 側のバッチサイズ・最大同時実行数・失敗時挙動で実運用の安定性が決まる。

重要性

DVA では「なぜ重複処理が起きるか」「どこで再試行されるか」の問われ方が多い。SQS 標準キューは少なくとも1回配送で重複があり得るため、アプリ側冪等性を必須で設計する。

使い分け

  • Partial batch response: 一部失敗のみ再試行したいとき
  • DLQ: 何度も失敗する毒メッセージ隔離
  • FIFO + message group: 厳密順序が必要なとき

誤解

「Lambda が失敗したら全件ロールバックされる」は誤解。バッチ処理では設定次第で成功分まで再配信される。部分失敗レスポンスを返さないと再処理コストが増える。

3分要点

① SQS→Lambda は ESM がポーリング実行。② 重複配送前提で idempotency key を持つ。③ Partial batch response を使わないと成功分まで巻き込まれやすい。④ 可観測性は CloudWatch Logs + ApproximateAgeOfOldestMessage で遅延把握。

Topic 2: Parameter Store vs Secrets Manager（設定と秘密情報の境界設計）

定義

AWS Systems Manager Parameter Store は構成値管理、AWS Secrets Manager は秘密情報（DB認証情報/APIキー等）管理に最適化されたサービス。どちらもKMS暗号化可。

重要性

DVA では「資格情報をコードに埋めない」「ローテーション要件があるなら何を選ぶか」が頻出。設計判断を問う実務寄り問題が多い。

使い分け

  • Parameter Store: 単純構成値、コスト重視、階層キー管理
  • Secrets Manager: 自動ローテーション、監査、RDS連携
  • Lambda: 初期化時キャッシュ + TTLでAPIコール削減

誤解

「SecureString なら Secrets Manager 不要」は半分正解。暗号化だけなら可能だが、自動ローテーション運用まで含めるなら Secrets Manager が本命。

3分要点

① 値の性質で分ける（設定値か秘密か）。② ローテーション要件の有無が分岐点。③ IAM最小権限は `ssm:GetParameter` と `secretsmanager:GetSecretValue` を資源ARNで限定。④ Lambdaでは毎invoke取得せずキャッシュ戦略を併用。
