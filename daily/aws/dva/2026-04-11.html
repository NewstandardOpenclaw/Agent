Topic 1: DynamoDB の条件付き書き込み（ConditionExpression）競合防止の要

定義

ConditionExpression は、指定条件を満たす場合だけ PutItem / UpdateItem / DeleteItem を成功させる仕組み。条件不成立なら ConditionalCheckFailedException で失敗する。

重要性

  • 同時更新が発生する API で「最後に書いた人勝ち」を避けられる。
  • DVA では楽観ロック（version 属性）や重複登録防止の問題が高頻度で出る。

使い分け

  • 新規作成のみ許可: attribute_not_exists(pk) で重複キー登録を防ぐ。
  • 更新競合を防ぐ: version = :expected 条件で楽観ロックを実装。
  • 在庫・残高の下限管理: stock >= :qty のように業務ルールをDB側で保証。

誤解

  • 「先に Get して確認すれば十分」→ 競合タイミングで破綻する。書き込み時条件が必要。
  • 「失敗したら即リトライでOK」→ 条件失敗はロジック衝突の可能性が高く、原因判定なしリトライは危険。

3分要点

  • 整合性は“読んで判断”より“書く時に条件保証”。
  • 重複防止・競合防止・業務制約の3用途を覚える。
  • ConditionalCheckFailedException は設計上の想定内エラーとして扱う。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Expressions.ConditionExpressions.html

Topic 2: Lambda の同時実行制御（Reserved / Provisioned Concurrency）遅延と過負荷を両制御

定義

Reserved Concurrency は関数の最大同時実行数を予約・制限する機能。Provisioned Concurrency は初期化済み実行環境を事前確保し、コールドスタートを抑える機能。

重要性

  • 下流DBを守るには上限管理（Reserved）が有効。
  • API応答の安定化には低遅延化（Provisioned）が有効。

使い分け

  • 突発トラフィックでDBが詰まる: Reserved で上限を設定し、バックプレッシャーを作る。
  • 朝9時に必ず低遅延: Provisioned をスケジュール増減してコスト最適化。
  • 重要API + 他関数の共存: 重要関数へ Reserved を割り当て、アカウント全体枯渇を防ぐ。

誤解

  • 「Provisioned を設定すれば無制限に捌ける」→ 同時実行数の上限は別途管理が必要。
  • 「Reserved は速くなる設定」→ 主目的は“制御”であり“高速化”ではない。

3分要点

  • Reserved = 守り（上限・隔離）、Provisioned = 攻め（低遅延）。
  • 試験では要件が「保護」か「応答速度」かを先に判定。
  • 両者は排他的ではなく、組み合わせで使う。

参照: https://docs.aws.amazon.com/lambda/latest/dg/configuration-concurrency.html

Topic 3: EventBridge リトライと DLQ 設計非同期障害を見逃さない

定義

EventBridge ルールはターゲット呼び出し失敗時に自動リトライでき、最終失敗イベントを DLQ（SQS） へ退避可能。これにより「失敗を消さない」非同期設計が実現できる。

重要性

  • Lambda/Step Functions 側の一時障害でイベントロストすると業務影響が大きい。
  • DVA では「再処理可能性」「監視容易性」を満たすアーキテクチャ選択が問われる。

使い分け

  • 一時的エラーが多い: リトライポリシーを有効にし、短期障害を自動吸収。
  • 恒久的エラーも発生: DLQ へ退避して手動/自動再処理フローを用意。
  • 運用で失敗検知を強化: DLQ メトリクスに CloudWatch Alarm を付与。

誤解

  • 「EventBridge は必ず1回届く」→ ターゲット失敗時の救済設計がないとロストしうる。
  • 「DLQ に入れたら完了」→ その後の再投入手順がないと“捨て置きキュー”になる。

3分要点

  • 非同期は成功経路より失敗経路の設計が差になる。
  • Retry + DLQ + Alarm を1セットで覚える。
  • 試験文で「durable」「recover」「operational visibility」を見たら候補。

参照: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rule-retry-policy.html
