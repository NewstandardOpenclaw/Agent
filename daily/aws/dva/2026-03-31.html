Topic 1: DynamoDB 条件付き書き込み（ConditionExpression）競合更新を防ぐ

定義

ConditionExpression は、指定条件が真のときだけ PutItem / UpdateItem / DeleteItem を成功させる機能。楽観ロックや重複登録防止の基盤になる。

重要性

  • 高トラフィック時でも「後勝ちで不整合が起きる」事故を防げる。
  • DVAで頻出の「同時更新」「冪等性」「重複排除」問題の中心論点。

使い分け

  • attribute_not_exists(PK): 初回作成のみ許可（重複作成防止）。
  • version = :v: 楽観ロック（更新競合検知）。
  • 在庫数/状態条件: ビジネス条件を満たす場合のみ更新。

誤解

  • 「DynamoDBは最後に書いた値が正しいので問題ない」→ 誤り。業務的には誤更新になる。
  • 「条件式エラーはシステム障害」→ 多くは想定内の競合で、再試行設計が必要。

3分要点

  • 同時更新がある設計では条件付き書き込みを前提にする。
  • 競合は「防ぐ」より「検知して安全に再試行」する。
  • 冪等性キーと組み合わせると実務で強い。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Expressions.ConditionExpressions.html

Topic 2: EventBridge + SQS で非同期疎結合を作る

定義

EventBridgeでイベントを受け、SQSキューに配送してコンシューマ（Lambda等）が後続処理する構成。プロデューサとコンシューマを時間的・実装的に分離できる。

重要性

  • 突発的な負荷ピークを吸収し、障害時の連鎖失敗を抑制できる。
  • DVAの「サーバレスで信頼性を高める設計」問題で定番。

使い分け

  • EventBridge: ルーティングとフィルタリング（どのイベントを誰に渡すか）。
  • SQS Standard: 高スループット優先、順序不要な処理。
  • SQS FIFO: 重複排除・順序保証が必須の処理。

誤解

  • 「EventBridgeだけで再試行・バッファは十分」→ 誤り。長時間滞留やバックプレッシャにはSQSが有効。
  • 「FIFOなら絶対重複ゼロ」→ 設計次第。MessageDeduplicationIdの扱いが重要。

3分要点

  • EventBridgeはイベント交通整理、SQSは処理の緩衝材。
  • 順序要件があるときだけFIFOを選ぶ。
  • DLQと可視性タイムアウトを必ずセットで設計する。

参照: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-targets.html

Topic 3: Cognito User Pool + API Gateway JWT オーソライザー

定義

Cognito User Poolで認証したJWTをAPI GatewayのJWTオーソライザーで検証し、正当ユーザーだけバックエンドへ通す認可方式。

重要性

  • Lambda内でトークン検証を毎回実装せず、認証責務を入口で統一できる。
  • DVAの「セキュアAPI」「マネージド認証」「最小実装」系でよく問われる。

使い分け

  • User Pool: エンドユーザー向け認証（サインアップ/サインイン）。
  • Identity Pool: AWSリソースへの一時認証情報払い出しが必要な場合。
  • API Gatewayオーソライザー: API単位でJWT検証とスコープ判定を実施。

誤解

  • 「認証できれば全APIにアクセス可」→ 誤り。スコープ/クレームで認可を分ける必要がある。
  • 「JWTは期限切れでも署名が正しければOK」→ 誤り。exp・iss・aud検証が必須。

3分要点

  • 認証（誰か）と認可（何ができるか）を分離して設計する。
  • API Gatewayで検証すると実装重複が減り、統制しやすい。
  • スコープ設計までやって初めて安全なAPIになる。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-jwt-authorizer.html
