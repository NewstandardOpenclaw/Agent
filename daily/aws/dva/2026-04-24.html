Topic 1: DynamoDB Accelerator（DAX）と DynamoDB 標準読み取りの使い分けレイテンシ最適化

定義

DAXは、DynamoDB向けのマネージド・インメモリキャッシュ。読み取りレイテンシをマイクロ秒単位まで短縮できる。

重要性

  • 高トラフィックAPIでp99遅延を下げる設計はDVAで頻出。
  • キャッシュをどこに置くか（アプリ内 / ElastiCache / DAX）の判断力が問われる。

使い分け

  • 同一キーへの読取集中: DAXが有効（キャッシュヒット率が高い）。
  • 強整合性が必須: DAX非対応ケースがあるため標準の強整合読取を選ぶ。
  • 書き込み主体: DAXの恩恵が小さいため、WCU設計やアクセスパターン見直しを優先。

誤解

  • 「DAXを入れれば全クエリが速くなる」→ ヒットしないキーは通常DynamoDB呼び出し。
  • 「DAXは整合性課題を自動解決する」→ アプリ側で許容する鮮度を設計する必要がある。

3分要点

  • “低遅延が必要な読み取り集中”にDAXを当てる。
  • 整合性要件が厳しい処理は通常読み取りを併用する。
  • DVA回答では、遅延要件と整合性要件を分けて説明する。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DAX.html

Topic 2: EventBridge 再試行・DLQ・Archive の設計イベント配信の堅牢化

定義

EventBridgeは、イベントルールでターゲット配信を行うサーバーレスイベントバス。失敗時の再試行、DLQ、Archive/Replayを組み合わせて信頼性を高める。

重要性

  • 「イベントが届かない」障害に対し、復旧経路（再実行）を準備できるかが実務・試験ともに重要。
  • 非同期システムの可観測性・保守性を決めるポイント。

使い分け

  • 一時的失敗: EventBridge既定再試行で吸収。
  • 継続失敗: DLQへ退避し、原因別に再投入フローを準備。
  • 仕様変更後の再処理: Archive + Replayで過去イベント再適用。

誤解

  • 「再試行があるのでDLQ不要」→ 永続失敗でイベントが失われるリスクがある。
  • 「Archiveはログ用途だけ」→ 実運用ではリプレイ復旧の中核になる。

3分要点

  • 再試行・DLQ・Replayを“復旧セット”として設計する。
  • 失敗イベントは可視化（メトリクス/アラート）まで含める。
  • DVAでは「失敗後にどう再実行するか」を明言する。

参照: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rule-retry-policy.html

Topic 3: Lambda エイリアス + CodeDeploy canary デプロイ安全な本番リリース

定義

Lambdaエイリアスはバージョン固定のトラフィックルーティング先。CodeDeployと組み合わせると canary / linear で段階的に新バージョンへ移行できる。

重要性

  • 障害時の影響範囲を小さくし、短時間でロールバックできる。
  • DVAのデプロイ問題では「ゼロダウンタイム」と「リスク低減」の両立が問われる。

使い分け

  • 変更規模が小さい: Canary 10%→100% で早く検証。
  • 変更規模が大きい: Linearで段階比率を細かく刻む。
  • 即時切替が必要: All-at-once。ただし事前検証を強化する。

誤解

  • 「エイリアスだけで自動ロールバックされる」→ CloudWatchアラーム連携が必要。
  • 「バージョン発行せずに運用しても問題ない」→ 再現性と切り戻し性が落ちる。

3分要点

  • 本番は“バージョン固定 + エイリアス運用”が基本。
  • 段階配信とアラーム連携で失敗時の自動保護を作る。
  • DVA回答は「配信方式 + 監視 + ロールバック条件」をセットで述べる。

参照: https://docs.aws.amazon.com/codedeploy/latest/userguide/deployment-groups-configure-advanced-options.html
