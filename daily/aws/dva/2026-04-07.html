Topic 1: Lambda 失敗時リトライ設計（同期/非同期/SQSトリガー）失敗戦略の基礎

定義

Lambda の失敗時挙動は呼び出し方式で異なる。同期呼び出しは呼び出し元がエラー処理、非同期呼び出しは Lambda サービス側が再試行、SQS イベントソースは可視性タイムアウトと再受信で再試行される。

重要性

  • DVA は「同じ Lambda でもトリガーで失敗時動作が変わる」点をよく問う。
  • 誤った前提でリトライすると、重複処理・メッセージ滞留・課金増加が起きる。

使い分け

  • API Gateway → Lambda（同期）: クライアントへ即時結果を返す必要がある処理。
  • EventBridge/SNS → Lambda（非同期）: 即時応答不要で耐障害性重視の処理。
  • SQS → Lambda: バッファリングしつつ後追い処理したいワークロード。

誤解

  • 「Lambda は常に同じ回数だけ再試行される」→ 誤り。呼び出し経路ごとに別設計。
  • 「DLQ を付ければ重複処理は防げる」→ 誤り。重複対策は冪等設計が本体。

3分要点

  • まずトリガー種別を確認してから失敗戦略を考える。
  • 再試行制御と冪等性はセットで設計する。
  • DVA では「どこが再試行主体か」を見抜けると強い。

参照: https://docs.aws.amazon.com/lambda/latest/dg/invocation-retries.html

Topic 2: Cognito User Pool と Identity Pool の責務分離認証とAWS権限付与

定義

User Pool はユーザー認証（サインアップ/サインイン/トークン発行）を担う。Identity Pool は認証済み（または未認証）ユーザーに対し、一時的な AWS 認証情報（IAM ロール）を払い出す。

重要性

  • 「ログインできる」と「S3にアクセスできる」は別問題で、試験でも分離して問われる。
  • モバイル/SPA で最小権限アクセスを実現する実務知識として頻出。

使い分け

  • 認証だけ必要: User Pool 単体（アプリ内APIへのアクセス制御中心）。
  • AWSリソースへ直接アクセス: User Pool + Identity Pool 連携。
  • ゲスト利用を許可: Identity Pool の未認証ロールを使い、権限を厳格に限定。

誤解

  • 「User Pool があれば AWS 資格情報も自動発行される」→ 誤り。Identity Pool が必要。
  • 「Identity Pool は認証機能を持つ」→ 誤り。認証結果を受けて権限を仲介する層。

3分要点

  • User Pool = 認証、Identity Pool = AWS 権限付与。
  • 試験で迷ったら「何を実現したいか（ログイン or AWSアクセス）」で切る。
  • 最小権限の IAM ロール設計まで含めて考える。

参照: https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-identity.html

Topic 3: CodeDeploy Blue/Green と Canary リリースの判断軸

定義

Blue/Green は旧環境と新環境を並行稼働させて切替える方式。Canary は一部トラフィックだけ新バージョンへ流し、問題なければ段階的に拡大する方式。

重要性

  • デプロイ失敗時の影響範囲をどこまで抑えるかは DVA の設計問題で定番。
  • 「最小ダウンタイム」「安全なロールバック」「コスト」のトレードオフ判断が問われる。

使い分け

  • Blue/Green: 切戻しの速さを最優先し、環境を二重持ちできる場合。
  • Canary: 実トラフィックでリスクを小さく検証しながら進めたい場合。
  • 組み合わせ: ECS/Lambda では段階配信 + CloudWatch Alarm 連動の自動停止が有効。

誤解

  • 「Blue/Green は必ず無停止」→ 誤り。データ移行や接続切替で瞬断は起こり得る。
  • 「Canary は必ず安全」→ 誤り。監視指標と自動ロールバック条件が弱いと検知漏れする。

3分要点

  • Blue/Green は切戻し速度、Canary は段階検証が強み。
  • 監視アラーム条件まで作って初めて“安全な配信”になる。
  • DVA は要件文の優先順位（可用性/コスト/速度）で選択肢を切る。

参照: https://docs.aws.amazon.com/codedeploy/latest/userguide/deployment-configurations.html
