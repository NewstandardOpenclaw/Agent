Topic 1: EventBridge の リトライ・DLQ・再処理設計イベントロス防止

定義

Amazon EventBridgeはイベントバス経由でサービス間を疎結合に接続する。ターゲット配信失敗時はリトライされ、回復不能なイベントはDLQ（通常SQS）へ退避できる。

重要性

  • 一時障害でイベントを捨てない設計ができるため、業務データの欠落を防げる。
  • DVAの「信頼性要件を満たす非同期設計」問題で高頻度に問われる。

使い分け

  • 一時的な依存先障害が多い: EventBridgeリトライ設定を活かす。
  • 失敗原因を後で検証したい: DLQへ退避し、再処理Lambdaを別系統で実行。
  • 重複処理が致命的: 下流を冪等化（イベントID保存など）して再試行を安全化。

誤解

  • 「EventBridgeがあるからDLQ不要」→ 恒久エラー時はDLQがないと追跡不能になりやすい。
  • 「再試行で必ず成功する」→ スキーマ不整合や権限不足は何度試しても失敗する。

3分要点

  • 非同期連携の要は「リトライ + DLQ + 冪等性」の3点セット。
  • 失敗イベントを可視化し、再処理経路をあらかじめ定義しておく。
  • DVAでは“失敗時の扱い”まで答えに含めると正答率が上がる。

参照: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-rule-retry-policy.html

Topic 2: Secrets Manager と Parameter Store の使い分け機密情報の安全運用

定義

AWS Secrets Managerはシークレットの保管と自動ローテーションに特化。Systems Manager Parameter Storeは設定値管理（SecureString含む）を行う。

重要性

  • 認証情報の漏えい防止だけでなく、運用時の更新負荷を下げられる。
  • DVAの「最小運用コストで安全性を高める選択」問題で狙われる論点。

使い分け

  • DB資格情報を定期更新したい: Secrets Manager（ローテーション機能）
  • アプリ設定値を階層管理したい: Parameter Store（パス構造で整理）
  • アクセス監査を厳密にしたい: どちらもIAM/KMS連携し、CloudTrailで追跡。

誤解

  • 「SecureStringならSecrets Managerと同じ」→ 自動ローテーションやシークレット連携機能が異なる。
  • 「環境変数に平文で置いても問題ない」→ 露出面が広がり、ローテーション運用も難しくなる。

3分要点

  • “更新頻度が高い機密情報”はSecrets Managerが第一候補。
  • “設定値中心”ならParameter Storeがシンプルで扱いやすい。
  • DVAでは要件（ローテーション有無）に紐づけて選ぶのが鉄則。

参照: https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html

Topic 3: API Gateway Usage Plan とスロットリング設計過負荷と不正利用の抑制

定義

Usage PlanはAPIキー単位でレート制限（Throttle）とクォータを適用する仕組み。API Gatewayステージ/メソッドに紐づけて利用量を制御できる。

重要性

  • 突発アクセスや特定クライアントの暴走からバックエンドを守れる。
  • DVAでの「可用性と公平性を保つAPI設計」の定番対策。

使い分け

  • 顧客プラン別に制限を変えたい: Usage Planを複数用意してAPIキーを分離。
  • 全体保護を優先: ステージ/メソッド単位のスロットリングで上限を設定。
  • 認証必須API: APIキーは利用制御、認証認可はCognito/IAM/Authorizerで別管理。

誤解

  • 「APIキーがあれば認証は不要」→ APIキーは本人確認ではなく利用量管理が主目的。
  • 「スロットリングを上げれば性能改善」→ バックエンド限界を超えると障害を拡大する。

3分要点

  • Usage Planは“誰にどれだけ使わせるか”を制御する手段。
  • 認証・認可と利用量制御は役割が違うので分離設計する。
  • DVAでは「保護対象（全体/顧客別）」に合わせた制限設計を選ぶ。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-api-usage-plans.html
