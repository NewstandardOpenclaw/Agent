Topic 1: DynamoDB 条件付き書き込みで同時更新を制御する

定義
  • ConditionExpression は「この条件を満たすときだけ書き込みを成功させる」DynamoDB の原子操作。
  • Optimistic Locking は version 属性を条件に使い、先に更新されたデータへの上書きを防ぐ設計。
重要性
  • DVA で頻出の「高並行アクセス時のデータ整合性」を、追加インフラなしで実現できる。
  • 二重決済・在庫過剰引当など、業務事故に直結するレースコンディションを抑止できる。
使い分け
  • 「存在しないときのみ作成」→ attribute_not_exists(pk) を使用。
  • 「前提バージョン一致時のみ更新」→ version = :expected を使用。
  • 複数アイテムの一貫性が必要 → TransactWriteItems で条件付き更新をまとめる。
よくある誤解
  • 「ConditionalCheckFailedException は障害」→ 多くは想定内の競合シグナル。再読込→再試行が正しい。
  • 「最後に書いた方が勝つで十分」→ 試験では業務要件（重複防止・整合性）があるなら失点対象。
3分要点
  1. DynamoDB は条件式で安全な同時更新制御ができる。
  2. 競合エラーは設計上の通常系として扱う。
  3. 複数レコード整合はトランザクションで補完する。

Topic 2: Lambda の冪等性（Idempotency）と再試行前提設計

定義
  • 冪等性 は、同じイベントが複数回処理されても結果が1回分に収束する性質。
  • Lambda は非同期再試行・SQS 再配信などで「重複実行が起こる前提」で設計する。
重要性
  • DVA の設計問題で「重複請求を防ぎたい」系はほぼ冪等性の知識を問っている。
  • 再試行を有効化したまま安全運用するための必須スキル。
使い分け
  • API 呼び出しなら Idempotency-Key（リクエストID）を保存して重複判定。
  • イベント処理なら eventId + TTL を DynamoDB で保持し短期重複を排除。
  • 部分失敗がある処理は「副作用前に記録」または「副作用を可逆化」して整合を保つ。
よくある誤解
  • 「Lambda は1回だけ実行される」→ 少なくとも1回（at-least-once）を前提にする。
  • 「DLQ を付ければ重複防止できる」→ DLQ は失敗退避であり、重複防止ロジックではない。
3分要点
  1. 再試行がある限り重複は起こる。
  2. 重複判定キーを外部ストアで管理する。
  3. DLQ は最後の避難先、冪等性とは役割が別。
