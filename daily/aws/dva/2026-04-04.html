Topic 1: Lambda 部分的バッチ応答（SQS イベントソース）再処理を最小化

定義

Partial Batch Response は、SQS トリガーの Lambda がバッチ内の失敗メッセージIDだけを返し、成功分を再配信対象から外す仕組み。全件失敗扱いを防げる。

重要性

  • 1件の例外でバッチ全体が再実行される無駄を削減できる。
  • 重複実行・DLQ流入・コスト増を同時に抑えやすい。

使い分け

  • 部分的バッチ応答を有効化: レコードごとに独立処理できるワークロード。
  • 全件失敗（従来）: レコード間依存が強く、整合性を一括で守る必要がある場合。
  • DLQ + リトライ制御: 恒久障害と一時障害を分離して運用したい場合。

誤解

  • 「部分的バッチ応答を使えば冪等性は不要」→ 誤り。少なくとも1回配信の前提は変わらない。
  • 「失敗IDを返せば可観測性は不要」→ 誤り。CloudWatch Logs/Metricsで失敗傾向を追う設計が必須。

3分要点

  • 目的は“失敗レコードだけ”を再試行すること。
  • 高スループットSQS連携ではコスト・遅延の改善効果が大きい。
  • DVAでは「再処理最小化」と「冪等性維持」の両立が問われる。

参照: https://docs.aws.amazon.com/lambda/latest/dg/with-sqs.html

Topic 2: DynamoDB 条件付き書き込み（ConditionExpression）

定義

ConditionExpression は、更新前に条件評価を行い、条件を満たすときだけ Put/Update/Delete を成功させる機能。楽観ロックや重複登録防止に使う。

重要性

  • 同時更新による上書き事故（Lost Update）をアプリ側で防げる。
  • 分散システムでの整合性制御を、追加ミドルウェアなしで実装できる。

使い分け

  • attribute_not_exists(pk): 新規作成の重複防止。
  • version 属性比較: 楽観ロック（競合検知）を実装。
  • TransactWriteItems: 複数アイテムを一括で原子的に守りたい場合。

誤解

  • 「DynamoDBは自動で競合解決してくれる」→ 誤り。要件に応じた条件式設計が必要。
  • 「ConditionalCheckFailedException は障害」→ 誤り。想定された競合のシグナルとして処理すべき。

3分要点

  • “条件を満たした時だけ書く”で整合性を守る。
  • 競合は例外で検知し、リトライ/ユーザー通知へ分岐する。
  • DVAでは version 管理と条件式の組み合わせが定番。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Expressions.ConditionExpressions.html

Topic 3: API Gateway Usage Plan と API Key の実運用

定義

Usage Plan は API Key ごとにレート制限（Throttle）とクォータ（Quota）を設定し、利用者単位で API 利用量を制御する仕組み。

重要性

  • 特定クライアントの過剰アクセスからバックエンドを保護できる。
  • SaaSのプラン別制御（Free/Pro等）をシンプルに実現しやすい。

使い分け

  • Usage Plan + API Key: 利用量管理・課金連動を行いたい。
  • Cognito/IAM 認可: ユーザー本人確認・権限制御を厳密にしたい。
  • AWS WAF: IP/パターンベースの攻撃トラフィックを前段で遮断したい。

誤解

  • 「API Key は認証そのもの」→ 誤り。本人性の保証には向かない。
  • 「Usage Plan があればDDoS対策は十分」→ 誤り。WAF/Shield等との多層防御が必要。

3分要点

  • Usage Plan は“誰がどれだけ使えるか”の管理機能。
  • 認証・認可とは役割が別。混同しない。
  • DVAでは「利用量制御」と「セキュリティ制御」の責務分離が得点ポイント。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-api-usage-plans.html
