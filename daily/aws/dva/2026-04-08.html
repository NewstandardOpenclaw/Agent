Topic 1: API Gateway の認可方式（IAM / Lambda Authorizer / Cognito）入口の守り方

定義

API Gateway では主に3方式でアクセス制御する。IAM認可は SigV4 署名で AWS プリンシパルを判定、Lambda Authorizerは独自ロジックでトークン検証、Cognito User Pool Authorizerは JWT を標準的に検証する。

重要性

  • DVAでは「利用者が誰か」に応じた最適認可方式を問う問題が頻出。
  • 認可方式を誤ると、実装コスト増か過剰権限のどちらかに寄りやすい。

使い分け

  • 社内ツールやAWS主体クライアント: IAM認可で統一しやすい。
  • 外部IdPや複雑判定: Lambda Authorizerで柔軟に制御。
  • モバイル/SPAの標準ログイン: Cognito Authorizerが最短。

誤解

  • 「Lambda Authorizerが常に最強」→ 誤り。要件次第で運用負荷が最大化する。
  • 「Cognitoを使えばIAM設計不要」→ 誤り。バックエンド側の権限分離は別問題。

3分要点

  • まずクライアント種別（AWS主体か一般ユーザーか）を確認。
  • 標準要件は Cognito、特殊要件のみ Lambda Authorizer。
  • DVAでは“最小実装で要件達成”を選ぶと正答率が上がる。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-control-access-to-api.html

Topic 2: DynamoDB 条件付き書き込みで競合を防ぐ同時更新の基本

定義

DynamoDB の ConditionExpression を使うと、「特定条件を満たす場合のみ Put/Update する」制御ができる。代表例は楽観ロック（version 属性一致時のみ更新）。

重要性

  • サーバレス構成では同時実行が前提で、単純上書きはデータ破損を招く。
  • DVAでは“整合性を保つ最小コストの方法”として条件付き更新がよく出る。

使い分け

  • 重複登録防止: attribute_not_exists(pk) で初回登録のみ許可。
  • 楽観ロック: version の一致を条件に更新し、失敗時は再読込。
  • 在庫減算: 残量 > 0 を条件にして負在庫を防止。

誤解

  • 「トランザクションを使わないと競合防止できない」→ 誤り。単一アイテムなら条件付き更新で十分。
  • 「ConditionalCheckFailedException は障害」→ 誤り。想定された競合シグナル。

3分要点

  • “書き込む前提条件”を式にしてDB側で判定させる。
  • 競合時はリトライ設計（指数バックオフ）までセット。
  • 試験では「最小変更で競合回避」がキーワード。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Expressions.ConditionExpressions.html

Topic 3: SQS FIFO の重複排除と順序保証の限界メッセージング設計

定義

SQS FIFO は MessageGroupId 単位で順序を保証し、DeduplicationId（または本文ハッシュ）で重複投入を一定時間抑制する。全体の厳密順序ではなく“グループ内順序”が本質。

重要性

  • DVAは「順序が必要」と書かれたら FIFO を選ぶだけでなく、並列性の制約まで問う。
  • 誤設計すると順序は守れてもスループット不足になりやすい。

使い分け

  • 厳密順序が必要な単位を分割可能: 複数 MessageGroupId で並列化。
  • 重複送信が起きやすい: 明示的 DeduplicationId を付与。
  • 高スループット最優先: 順序不要なら Standard を検討。

誤解

  • 「FIFO ならキュー全体で完全順序」→ 誤り。順序保証はグループ単位。
  • 「重複排除で完全に二重処理ゼロ」→ 誤り。アプリ側冪等性は依然必要。

3分要点

  • FIFOの設計肝は MessageGroupId の切り方。
  • 順序・重複排除・性能の3軸で要件を分解する。
  • DVAでは“順序範囲”を正しく読むと選択肢を削れる。

参照: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/FIFO-queues-understanding-logic.html
