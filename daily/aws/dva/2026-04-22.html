Topic 1: API Gateway レスポンスキャッシュと Cache-Control の整合設計遅延削減と整合性の両立

定義

API Gateway レスポンスキャッシュは、同一リクエストに対するバックエンド呼び出し結果を一定時間保持し再利用する機能。HTTPヘッダの Cache-Control とTTL設計を合わせることで、配信結果の鮮度を管理する。

重要性

  • 高頻度読み取りAPIのレイテンシとLambda実行回数を同時に削減できる。
  • DVAの設計問題で問われる「パフォーマンス最適化 + コスト抑制」の定番手段。

使い分け

  • 更新頻度が低いデータ: API GatewayキャッシュTTL長め（例: 300秒）。
  • 更新が頻繁なデータ: TTL短め + 必要に応じてキャッシュ無効化APIを実装。
  • クライアント側の再利用: Cache-Control/ETagで端末・CDN側キャッシュも活用。

誤解

  • 「キャッシュを有効化すれば常に正しい最新値が返る」→ TTL期間中は古い値が返る可能性がある。
  • 「キャッシュは無料」→ API Gatewayキャッシュは専用容量課金で、サイズ設計が必要。

3分要点

  • 目的は“高速化”だけでなく“バックエンド負荷平準化”。
  • TTLはビジネス許容遅延から逆算して決める。
  • DVAでは「鮮度要件とコスト要件のトレードオフ説明」が得点源。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-caching.html

Topic 2: EventBridge ルールのイベントパターン最適化不要起動を減らす

定義

EventBridgeイベントパターンは、受信イベントのフィールド条件でターゲット起動可否を判定するフィルタ。source/detail-type/detailなどを組み合わせ、必要イベントだけを流す。

重要性

  • LambdaやStep Functionsの無駄起動を防ぎ、コストとノイズを同時に削減できる。
  • イベント駆動設計での“誤発火”は障害に直結するため、試験でも実務でも重要。

使い分け

  • 広く拾う初期段階: source/detail-type中心の粗い条件。
  • 本番運用: detailの属性（status, region, resourceId）まで絞り込む。
  • 複数業務で共有: ルール分割して責務ごとにターゲット分離。

誤解

  • 「後段Lambdaでif判定すれば十分」→ 先にEventBridgeで絞る方がコスト・保守性ともに有利。
  • 「1ルールに全部詰め込むほど管理しやすい」→ 変更影響が広がり、障害時の切り分けが難しくなる。

3分要点

  • イベントパターンは“最初の防波堤”。
  • 条件は業務責務単位で分離すると運用しやすい。
  • DVAでは「不要実行削減 = コスト最適化」を明示する。

参照: https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-event-patterns.html

Topic 3: DynamoDB のパーティションキー設計とホットパーティション回避スケール障害を未然に防ぐ

定義

ホットパーティションは、特定パーティションキーにアクセスが集中してスループット上限へ達する状態。DynamoDBではキー分散設計が性能安定性の核心になる。

重要性

  • アプリ側で再試行が増え、遅延・タイムアウト・コスト増を招く主要原因になる。
  • DVAで頻出の「高トラフィック時に性能が不安定」問題の最重要論点。

使い分け

  • 偏りやすいID: ランダムサフィックス/バケット化で書き込み分散。
  • 時系列検索が必要: PKを分散し、SKにtimestampを持たせ範囲取得。
  • 読み込み最適化: GSIで参照パターン別にアクセス経路を分離。

誤解

  • 「オンデマンド課金ならホットパーティションは起きない」→ 起きる。物理分散制約は残る。
  • 「後でAuto Scalingが吸収してくれる」→ 単一キー集中はスケール設定だけでは解決しない。

3分要点

  • DynamoDBの成否はアクセスパターン先行設計で決まる。
  • “均等分散できるPK”を最初に決めるのが最重要。
  • DVA回答は「キー設計→GSI→再試行/監視」の順で述べると強い。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/bp-partition-key-design.html
