Topic 1: Lambda の失敗時リトライ設計（同期/非同期/イベントソース）失敗前提の設計

定義

Lambda の再試行は呼び出し方式で挙動が異なる。同期呼び出し（API Gateway など）は呼び出し元が制御し、非同期呼び出し（EventBridge など）は Lambda サービスが再試行、イベントソースマッピング（SQS/Kinesis）はソース側ポーリングで再処理される。

重要性

  • DVAでは「どこで再試行されるか」を誤ると、DLQ設定やタイムアウト設計を間違える。
  • 本番障害は“再試行の重複”か“再試行されない想定漏れ”のどちらかが多い。

使い分け

  • 即時応答API: 同期 + 呼び出し元でエラーハンドリング。
  • 疎結合イベント処理: 非同期 + on-failure destination を設定。
  • バッチ/キュー処理: SQS イベントソース + 可視性タイムアウト調整。

誤解

  • 「Lambda は常に自動リトライする」→ 誤り。同期呼び出しでは自動再試行しない。
  • 「DLQ を付ければ全失敗を回収できる」→ 誤り。経路ごとに destination 対応が必要。

3分要点

  • まず呼び出しモデルを判定（同期/非同期/イベントソース）。
  • 再試行責任の所在に合わせて監視・DLQ を置く。
  • DVAは「最小構成で失敗を取りこぼさない設計」が正解になりやすい。

参照: https://docs.aws.amazon.com/lambda/latest/dg/invocation-retries.html

Topic 2: API Gateway ステージ変数と Lambda エイリアス連携安全な段階リリース

定義

API Gateway のステージ変数を使うと、同じ API 定義でも環境ごとに呼び先を切り替えられる。Lambda では関数バージョンに対する Alias（例: prod, canary）を使い、段階的リリースを安全に行う。

重要性

  • 本番事故の多くはデプロイ切替手順の複雑さ由来。エイリアス活用でロールバックが速くなる。
  • DVAの設問は「最小変更で環境差分を管理」が繰り返し出る。

使い分け

  • dev/stg/prod の分離: ステージ変数で Alias 名を渡して切替。
  • 部分リリース: Lambda Alias の weighted routing でカナリア配信。
  • 即時ロールバック: Alias ポインタを前バージョンへ戻す。

誤解

  • 「ステージごとに API を複製するのが正道」→ 誤り。管理コストが高騰しやすい。
  • 「Alias は名前ラベルだけ」→ 誤り。トラフィック分割と運用制御の中核。

3分要点

  • 環境差分はステージ変数、コード差分は Lambda バージョンで管理。
  • 切替は Alias で行うとリスクが局所化される。
  • DVAでは「早く戻せる設計」を選ぶと失点しにくい。

参照: https://docs.aws.amazon.com/lambda/latest/dg/configuration-aliases.html

Topic 3: DynamoDB TTL と Streams を使った後処理自動化データ寿命管理

定義

DynamoDB の TTL は有効期限を過ぎたアイテムを自動削除する機能。削除イベントは DynamoDB Streams で補足でき、Lambda 連携により監査ログ生成や集計更新などの後処理を自動化できる。

重要性

  • 期限付きデータ（セッション・一時トークン）を手動削除すると運用漏れが起きる。
  • DVAでは「低運用コストでデータクリーンアップ」が典型テーマ。

使い分け

  • 期限切れ放置を防ぐ: TTL で自動削除。
  • 削除をトリガーに通知: Streams + Lambda でイベント処理。
  • 厳密な時刻削除が必要: TTL は遅延ありのため補助ジョブを併用。

誤解

  • 「TTL は指定時刻に必ず即削除」→ 誤り。削除はベストエフォートで遅延しうる。
  • 「TTL 削除は追跡できない」→ 誤り。Streams で監査用途に活用可能。

3分要点

  • TTL は“正確な秒単位制御”ではなく“運用簡素化”の機能。
  • 後続アクションが必要なら Streams 連携を前提に設計。
  • DVAでは「完全精度より managed 機能優先」が鍵になる。

参照: https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/TTL.html
