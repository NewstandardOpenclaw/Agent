Topic 1: Lambda 部分バッチ失敗レスポンス（SQSイベント）再処理を最小化

定義

SQSトリガーLambdaで、成功したメッセージは削除し、失敗したメッセージIDのみ batchItemFailures で返す仕組み。全件リトライではなく失敗分だけ再処理できる。

重要性

  • 1件エラーで10件まとめて再実行される無駄を防ぎ、コストと遅延を削減できる。
  • DVA頻出の「SQS + Lambda 信頼性設計」で高確率で問われる実務ポイント。

使い分け

  • 部分バッチ失敗ON: 1件ずつ成否が分かる処理（注文・通知・変換など）。
  • 全件失敗: バッチ全体の整合性が必要な処理（集計単位がバッチ固定など）。
  • DLQ併用: 恒久エラー（不正データ）は再試行ループを止める。

誤解

  • 「Lambdaが例外を投げれば失敗分だけ再処理される」→ 誤り。未設定だとバッチ全体再試行になりがち。
  • 「部分失敗を使えば冪等性は不要」→ 誤り。重複配信は依然発生し得るため冪等性は必須。

3分要点

  • SQS-Lambdaは“少なくとも1回”配信前提で設計する。
  • 部分バッチ失敗 + 冪等性 + DLQ が実務の基本セット。
  • 試験では「再処理を最小化しつつ信頼性確保」がキーワード。

参照: https://docs.aws.amazon.com/lambda/latest/dg/with-sqs.html

Topic 2: API Gateway usage plan と API key の役割分離

定義

Usage Plan はAPI利用量を契約単位で制御する仕組み。API keyは利用者識別に使い、認証・認可そのものはIAM/Cognito/Lambda authorizerなどで実施する。

重要性

  • パートナーAPIで「レート制限・クォータ管理」を簡潔に実装できる。
  • DVAで狙われる「API keyを認証目的で使ってはいけない」論点の中核。

使い分け

  • API key + usage plan: 利用量制御・課金区分・契約プラン差分。
  • Cognito/JWT: エンドユーザー認証が必要な公開API。
  • IAM SigV4: AWS内/サーバー間APIで厳密に権限制御したい場合。

誤解

  • 「API keyがあれば安全」→ 誤り。漏えい時に不正利用されるため、認証の代替にならない。
  • 「Usage planで厳密上限を必ず守れる」→ 誤り。ベストエフォート制御のため設計時は余裕を見る。

3分要点

  • API keyは“誰の利用か”を識別するタグ寄り機能。
  • アクセス可否は別レイヤーで認証・認可する。
  • 試験ではセキュリティ要件と利用量要件を分けて判断する。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-api-usage-plans.html

Topic 3: CodeBuild のローカルキャッシュ戦略（依存関係/ Docker layer）

定義

CodeBuildの local cache を使い、依存ライブラリやDockerレイヤーをビルドホスト上に再利用して、CI時間を短縮する最適化手法。

重要性

  • Node/Python/Java依存が重いプロジェクトではビルド時間を大きく削減できる。
  • DVAでは「最小変更でビルド高速化」の設問でよく登場する。

使い分け

  • LOCAL_SOURCE_CACHE: ソース再取得を短縮。
  • LOCAL_CUSTOM_CACHE: npm/pip/mavenキャッシュの再利用。
  • LOCAL_DOCKER_LAYER_CACHE: Docker buildのレイヤー再利用（特に有効）。

誤解

  • 「常にキャッシュが残る」→ 誤り。ホスト再割当てで失われることがある。
  • 「キャッシュすれば再現性も向上する」→ 誤り。固定バージョン管理が別途必要。

3分要点

  • 最初に効くのは依存キャッシュとDocker layer cache。
  • 再現性は lock file で担保、速度は cache で担保と切り分ける。
  • 試験では「最小運用負荷で高速化」を選ぶのが正解になりやすい。

参照: https://docs.aws.amazon.com/codebuild/latest/userguide/build-caching.html
