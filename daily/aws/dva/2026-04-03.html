Topic 1: API Gateway Lambda オーソライザー（JWT/Cognito）認可ロジックを分離

定義

API Gateway のLambda オーソライザーは、受信トークンを検証し、アクセス可否を IAM ポリシーとして返す仕組み。HTTP API では JWT オーソライザー、REST API では Lambda オーソライザーを使う場面が多い。

重要性

  • 認証・認可の責務をバックエンド本体から分離できる。
  • DVA で頻出の「最小変更でセキュリティ強化」問題に直結する。

使い分け

  • JWT オーソライザー: 標準的な OIDC/JWT 検証を高速に実装したい。
  • Lambda オーソライザー: 独自ヘッダー・外部 RBAC 判定など柔軟性が必要。
  • Cognito User Pool + JWT: マネージド認証を優先し運用負荷を下げたい。

誤解

  • 「オーソライザーを入れればバックエンド側の権限チェックは不要」→ 誤り。リソース単位の最終防衛は必要。
  • 「Lambda オーソライザーは常に最適」→ 誤り。要件が標準 JWT なら過剰実装になる。

3分要点

  • 標準要件は JWT オーソライザー、独自要件は Lambda オーソライザー。
  • 認可判定は API 手前で実施し、アプリ本体を薄く保つ。
  • 試験では「運用負荷が低いマネージド選択」が高得点になりやすい。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/http-api-jwt-authorizer.html

Topic 2: SQS FIFO の重複排除と順序保証

定義

SQS FIFO は MessageGroupId 単位で順序を保証し、MessageDeduplicationId（またはコンテンツベース重複排除）で重複受信を抑えるキュータイプ。

重要性

  • 注文処理・在庫更新など「順番が意味を持つ処理」で整合性を守れる。
  • 「少なくとも1回配信」前提でも、アプリ設計で重複事故を減らせる。

使い分け

  • FIFO: 厳密な順序制御・重複抑止が必須。
  • Standard: スループット優先で順序要件が緩い。
  • 複数 MessageGroupId: FIFO のまま並列度を確保したい場合。

誤解

  • 「FIFO なら重複は100%起きない」→ 誤り。設計上の冪等性は引き続き必要。
  • 「MessageGroupId は固定値が正解」→ 誤り。固定すると並列処理が極端に落ちる。

3分要点

  • 順序が必要なら FIFO、不要なら Standard が基本。
  • FIFO では GroupId 設計が性能の鍵。
  • DVA では「順序 + 重複排除 + スケール」のトレードオフを問われる。

参照: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/FIFO-queues.html

Topic 3: CodeDeploy Blue/Green（ECS/Lambda）のロールバック設計

定義

Blue/Green デプロイは、新旧環境を並行で保持し、トラフィック切替で新バージョンへ移行する手法。異常時は旧環境へ即時ロールバック可能。

重要性

  • 本番障害時の復旧時間（MTTR）を短縮できる。
  • DVA の「可用性を維持したデプロイ戦略」問題で鉄板。

使い分け

  • Blue/Green: 影響最小で安全に切替したい。
  • Canary: 段階リリースでリスクを観測しながら進めたい。
  • In-place: 環境制約が強く、停止許容できる内部システム。

誤解

  • 「Blue/Green なら監視設定は不要」→ 誤り。CloudWatch Alarm 連携がロールバック自動化の前提。
  • 「切替後すぐ旧環境を破棄してよい」→ 誤り。検証窓を残す運用が安全。

3分要点

  • 安全性重視なら Blue/Green、段階検証なら Canary。
  • 自動ロールバックは Alarm 設計まで含めて完成。
  • 試験では「ダウンタイム最小 + 失敗時即復旧」を満たす選択が正解になりやすい。

参照: https://docs.aws.amazon.com/codedeploy/latest/userguide/deployment-groups-configure-advanced-options.html
