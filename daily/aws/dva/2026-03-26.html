Topic 1: DynamoDB ConditionExpression で「上書き事故」を防ぐ

定義
  • ConditionExpression は PutItem/UpdateItem/DeleteItem の実行条件を指定する式。条件を満たさないと ConditionalCheckFailedException になる。
  • 代表例は attribute_not_exists(PK)（重複作成防止）や version = :v（楽観ロック）。
重要性
  • DVAで頻出の「同時更新」「重複登録」「冪等性」対策を、アプリ側のif文ではなくDB原子操作で実現できる。
  • サーバレス構成（API Gateway + Lambda）では同時実行が前提なので、条件付き書き込みの有無が設計品質を分ける。
使い分け
  • 初回作成のみ許可したい → PutItem + attribute_not_exists。
  • 更新競合を検出したい → version属性で楽観ロック。
  • 複数アイテムを一貫条件で処理したい → TransactWriteItems + ConditionCheck。
よくある誤解
  • 「Lambdaで先にGetして判定すれば十分」→ その間に別リクエストが更新すると競合する（TOCTOU）。
  • 「条件失敗はエラーだからリトライで潰す」→ 競合検知の正常系であり、業務上の再入力/再取得フローが必要。
3分要点
  1. 同時更新対策は ConditionExpression を第一候補にする。
  2. 重複防止は attribute_not_exists、競合検知は version 比較。
  3. 条件失敗は「設計どおりの拒否」なのでハンドリング方針を決める。

Topic 2: Lambda の非同期呼び出し失敗先（DLQ vs On-failure Destination）

定義
  • 非同期呼び出し（Event）では Lambda が内部キューで再試行し、最終失敗時に行き先を設定できる。
  • DLQ は失敗イベント本体を SQS/SNS に送る機能、On-failure Destination は実行結果メタデータを含むレコードを SQS/SNS/EventBridge/Lambda に送る機能。
重要性
  • DVAでは「失敗イベントをどう観測し、どう再処理するか」の選択問題が出やすい。
  • どこまで詳細な失敗情報が必要かで選択が変わるため、単なる用語暗記ではなく運用意図の理解が必要。
使い分け
  • 失敗イベントをそのまま再投入したい → DLQ。
  • 失敗理由やリクエストコンテキストも分析したい → On-failure Destination。
  • 失敗を運用イベントとしてルーティングしたい → Destination + EventBridge で通知/自動修復へ展開。
よくある誤解
  • 「DLQとDestinationは同じ」→ 格納内容・連携先・ユースケースが違う。
  • 「再試行回数は無限」→ 非同期は最大再試行回数/イベント有効期限の設定に従う。
3分要点
  1. DLQは再処理寄り、Destinationは可観測性・連携寄り。
  2. 失敗後のオペレーション設計（誰が何をトリガーするか）まで決める。
  3. DVAでは「必要情報量」と「連携先」で選ぶと正答しやすい。
