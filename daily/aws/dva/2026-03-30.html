Topic 1: API Gatewayのリクエスト検証（Request Validator）不正入力の入口遮断

定義

Request Validator は、API Gatewayで受信したパラメータやボディをモデル（JSON Schema）に照らして検証し、要件外リクエストをLambda到達前に弾く機能。

重要性

  • Lambdaで毎回バリデーションを書く重複実装を減らし、失敗を早期に返せる。
  • DVA試験で頻出の「最小工数でAPI品質を上げる設計」問題に直結する。

使い分け

  • API Gateway検証: 形式チェック（必須項目・型・最小長）を入口で統一。
  • Lambda側検証: 業務ルール（在庫数や契約状態など）を判定。
  • 両方併用: 形式は入口、業務はアプリで責務分離するのが実務的。

誤解

  • 「Gateway検証があればLambdaの検証は不要」→ 誤り。業務整合性は別問題。
  • 「検証エラーは500になる」→ 通常は4xxで返し、運用側で原因を追える。

3分要点

  • 入力品質はAPI入口で一定化する。
  • 形式検証と業務検証を分離すると保守が楽になる。
  • コスト削減（無駄なLambda実行抑止）にも効く。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-method-request-validation.html

Topic 2: Lambda環境変数 + KMS 暗号化の実務設計

定義

Lambda環境変数は設定値注入の標準手段。機微情報はKMSキーで暗号化し、復号権限をIAMで制御して安全に扱う。

重要性

  • コードに秘密情報を埋め込まず、デプロイと設定変更を分離できる。
  • DVAの「セキュアな構成」「最小権限」「運用性」の横断問題で問われる。

使い分け

  • 通常設定値: 環境変数で十分（例: ログレベル、タイムアウト閾値）。
  • 機密性が高い値: Secrets Manager / Parameter Store SecureStringを優先。
  • KMS権限: 実行ロールに必要最小限のkms:Decryptのみを許可。

誤解

  • 「KMS暗号化していれば誰でも安全に使える」→ 誤り。復号権限管理が本体。
  • 「環境変数は常に最適」→ 大量シークレット管理には向かない場合がある。

3分要点

  • 設定値とコードの分離はLambda運用の基本。
  • 秘密情報は専用ストア + IAM最小権限が原則。
  • KMSは“暗号化”だけでなく“誰が復号できるか”まで設計する。

参照: https://docs.aws.amazon.com/lambda/latest/dg/configuration-envvars.html

Topic 3: CodeDeploy Lambdaデプロイ（Canary/Linear）と自動ロールバック

定義

Lambdaのエイリアスに対し、CodeDeployで段階的にトラフィックを新バージョンへ移すデプロイ方式。CloudWatchアラーム連携で異常時に自動ロールバック可能。

重要性

  • 一括切替の事故リスクを下げ、影響範囲を小さくできる。
  • DVAの「可用性を維持しながら安全にリリース」系の典型テーマ。

使い分け

  • Canary: 少量トラフィックで短時間検証してから全量切替。
  • Linear: 定期的に段階増加、負荷を見ながら慎重に進める。
  • AllAtOnce: リスク低い小規模変更に限定して採用。

誤解

  • 「デプロイ成功＝運用成功」→ 誤り。アラーム閾値設計が未熟だと検知漏れする。
  • 「段階配信ならテスト不要」→ 誤り。事前テストと監視の両輪が必要。

3分要点

  • 段階配信は障害半径を抑えるための実装戦略。
  • CloudWatchアラームと自動ロールバックを必ずセットにする。
  • Canary/Linearの選定は変更リスクと監視成熟度で決める。

参照: https://docs.aws.amazon.com/codedeploy/latest/userguide/deployment-configurations.html
