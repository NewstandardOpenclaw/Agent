Topic 1: Cognito User Pool + API Gateway Cognito Authorizer のトークン検証設計認証不備を設計で潰す

定義

Cognito Authorizerは、API GatewayがJWT（主にID/Accessトークン）を受け取り、署名・有効期限・issuer/audienceを検証してAPIアクセス可否を判定する仕組み。

重要性

  • Lambda内で毎回トークン検証を書く必要が減り、認証漏れや実装ブレを防げる。
  • DVAでは「どこで認証/認可を担保するか」の境界設計が頻出。

使い分け

  • ユーザー属性に基づく認可: Cognitoグループ/クレームを使い、バックエンドで細粒度制御。
  • 機械間アクセス: User PoolよりIAM SigV4やCognito Identity Poolを検討。
  • 短命アクセストークン: 失効リスクを下げる代わりにリフレッシュ設計を必須化。

誤解

  • 「IDトークンとAccessトークンは同じ用途」→ API保護は通常Accessトークン中心。
  • 「Authorizerを入れれば認可も自動で完了」→ リソース単位の権限判定は別途設計が必要。

3分要点

  • 認証はAPI Gatewayで前段ブロック、業務認可はバックエンドで補完。
  • トークン寿命・更新フロー・クレーム設計をセットで考える。
  • DVA回答では「検証場所」と「責務分離」を明示すると強い。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-integrate-with-cognito.html

Topic 2: SQS + Lambda の部分バッチ失敗（Partial Batch Response）活用再試行コストを最小化

定義

Partial Batch Responseは、SQSイベントのバッチ処理で失敗したメッセージIDのみを返し、成功分の再処理を回避するLambda連携機能。

重要性

  • 1件失敗でバッチ全体再試行になる無駄を防ぎ、処理時間と料金を大幅に削減。
  • イベント駆動の信頼性設計（再試行・DLQ）はDVAの鉄板出題領域。

使い分け

  • 失敗が局所的: Partial Batch Responseで失敗IDだけ再試行。
  • 恒久失敗が多い: maxReceiveCountを低めにしてDLQ退避を早める。
  • 順序が必須（FIFO）: メッセージグループ単位のブロッキング影響を評価する。

誤解

  • 「可視性タイムアウトを長くすれば再試行設計は不要」→ 失敗分類とDLQ設計は別問題。
  • 「DLQは最後の保管庫なので監視不要」→ 再投入フローとアラートがないと運用停止する。

3分要点

  • 再試行は“全部”ではなく“失敗分だけ”が基本方針。
  • Visibility Timeoutは処理時間の上限に合わせる。
  • DVAでは「Partial Batch + DLQ + 監視」をセットで答える。

参照: https://docs.aws.amazon.com/lambda/latest/dg/services-sqs-errorhandling.html

Topic 3: KMS キーポリシーと IAM ポリシーの評価順序暗号化の権限境界を理解する

定義

AWS KMSでは、暗号化API呼び出し可否が「キーポリシー + IAMポリシー + 必要に応じてグラント」の組み合わせで評価される。どれかが欠けると利用できないケースがある。

重要性

  • アプリ本番で起きる「IAM許可したのにDecryptできない」障害の原因を最短で切り分けられる。
  • DVA試験でのセキュリティ問題は、実行ロールとKMS権限の整合確認が核心。

使い分け

  • 同一アカウント内の標準運用: キーポリシーでIAM利用を有効化し、IAM側で最小権限化。
  • クロスアカウント: 共有先プリンシパルをキーポリシーに明示し、相手側IAMも許可。
  • 一時利用: グラントで期限付きアクセスを発行。

誤解

  • 「IAMでAllowなら必ずKMS利用可能」→ キーポリシー側の許可が不足していると失敗する。
  • 「AWS管理キーなら設定不要」→ 利便性は高いが、厳密なアクセス境界が必要な要件には不向きな場合がある。

3分要点

  • KMSは“鍵側の許可”を必ず確認する。
  • 権限トラブルは CloudTrail の KMSイベントで追跡する。
  • DVA回答は「誰が・どの鍵で・どの操作を」三点で整理する。

参照: https://docs.aws.amazon.com/kms/latest/developerguide/control-access.html
