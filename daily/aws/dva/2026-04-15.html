Topic 1: API Gateway Lambda オーソライザーのキャッシュ戦略認証レイテンシ最適化

定義

Lambda オーソライザーは、API呼び出し前にトークン検証を実行する仕組み。キャッシュTTLを設定すると、同一条件の認可結果を一定時間再利用できる。

重要性

  • 毎リクエストで認証Lambdaを起動しないため、遅延とコストを同時に削減できる。
  • DVA頻出の「認証は維持しつつ性能を上げる」要件に直結する。

使い分け

  • 短寿命トークン/即時失効が重要: TTLを短めに設定。
  • 高トラフィックAPIで負荷削減優先: TTLを長めに設定してヒット率を上げる。
  • ユーザー別権限差が大きい: キャッシュキー設計（トークンやヘッダー）を厳密にする。

誤解

  • 「キャッシュすれば常に安全」→ 失効反映が遅れる可能性があるためTTL設計が必須。
  • 「IAM認証と同じ挙動」→ IAM署名認証とLambdaオーソライザーは責務と運用ポイントが異なる。

3分要点

  • オーソライザーは“正しさ”と“速さ”のトレードオフ設計。
  • TTLはセキュリティ要件（失効即時性）から逆算する。
  • DVAでは「要件に応じたTTL/方式選択」が正答の鍵。

参照: https://docs.aws.amazon.com/apigateway/latest/developerguide/apigateway-use-lambda-authorizer.html

Topic 2: SQS FIFO の MessageGroupId と重複排除順序保証と二重処理防止

定義

FIFOキューは順序保証と重複排除を提供するSQS。MessageGroupId単位で順序が守られ、MessageDeduplicationIdまたはContent-based deduplicationで重複送信を抑止する。

重要性

  • 注文処理や状態遷移など、順序が壊れると不整合を起こす処理に有効。
  • DVAで問われる「Exactly-onceに近づける設計」の中核知識。

使い分け

  • 顧客IDごとに順序保証: 顧客IDをMessageGroupIdにする。
  • 同一イベント再送が多い: 明示的DeduplicationIdで重複抑止。
  • 処理並列性を上げたい: Groupを細かく分割して並列度を確保。

誤解

  • 「FIFOなら全体で1本の順序」→ 順序はGroup単位。Groupが違えば並列実行される。
  • 「FIFOなら完全に重複ゼロ」→ 重複排除ウィンドウ外や下流再試行で重複対策は依然必要。

3分要点

  • 順序保証の境界はMessageGroupIdで決まる。
  • 重複排除は“送信時の保護”、下流側の冪等性もセットで設計する。
  • 試験では「順序+スケール」の両立案を選べるかが勝負。

参照: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/FIFO-queues.html

Topic 3: Lambda デプロイの段階的トラフィックシフト（CodeDeploy）リスク最小の本番反映

定義

CodeDeploy for Lambdaは、新バージョンへトラフィックを段階的に切り替えるデプロイ方式。CloudWatchアラーム連携で異常時に自動ロールバック可能。

重要性

  • 全量即時切替の障害インパクトを防ぎ、可用性を維持できる。
  • DVAで定番の「安全なデプロイ方式」問題にそのまま使える。

使い分け

  • 影響大の変更: Canary/Linearで段階反映。
  • 即時反映が必要だが監視必須: All-at-once + 厳密なアラーム。
  • 失敗許容度が低い: PreTraffic/PostTrafficフックで検証を挟む。

誤解

  • 「Alias更新だけで十分安全」→ 監視・自動戻しがないと事故時対応が遅れる。
  • 「段階反映ならテスト不要」→ 事前テスト不足は段階反映でも防げない。

3分要点

  • 安全なLambdaデプロイは「段階反映 + 監視 + 自動ロールバック」。
  • トラフィックシフト戦略は変更リスクと業務影響で選ぶ。
  • DVAでは“最小ダウンタイムで障害半径を小さく”が正解パターン。

参照: https://docs.aws.amazon.com/codedeploy/latest/userguide/deployment-configurations.html
