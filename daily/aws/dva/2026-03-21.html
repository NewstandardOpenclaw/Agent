Topic 1: Lambda SnapStart と Provisioned Concurrency の使い分け

定義
  • SnapStart: Lambda 関数の初期化済みスナップショットを使い、コールドスタート時間を短縮する機能（Java中心で有効性が高い）。
  • Provisioned Concurrency: 事前に実行環境をウォーム状態で確保しておく機能。予測可能な低レイテンシを提供。
重要性
  • DVA では「低レイテンシ要件への実装選択」が頻出。コールドスタート対策の違いを問われる。
  • コスト・運用・言語依存の観点をセットで判断できると正答率が上がる。
使い分け
  • ピーク時間が明確、常時低遅延が必要: Provisioned Concurrency優先。
  • Java関数の起動短縮を低コストで狙う: SnapStartが有力。
  • トラフィック変動が大きく、常時確保コストを避けたい: まず SnapStart 検討。
よくある誤解
  • 「SnapStart = すべてのランタイムで同じ効果」→ ランタイム依存がある。
  • 「Provisioned Concurrency は常に最安」→ 常時確保分の料金が発生する。
  • 「両者は完全に同じ目的」→ どちらも起動遅延対策だが、実装前提と運用モデルが異なる。
3分要点
  1. 「絶対に安定低遅延」なら Provisioned Concurrency。
  2. 「Java の初期化が重い」なら SnapStart を第一候補。
  3. 試験では、性能要件 + コスト要件 + ランタイム条件の3点で選ぶ。

Topic 2: API Gateway の Lambda オーソライザー vs Cognito オーソライザー

定義
  • Lambda オーソライザー: 独自ロジックで認可を実装。トークン/リクエスト情報を柔軟に判定。
  • Cognito オーソライザー: Cognito User Pool の JWT 検証を標準で利用する方式。
重要性
  • DVA では「最小実装で安全に認証/認可を実現する」問題が多い。
  • 運用負荷を下げる選択（マネージド優先）が正解になりやすい。
使い分け
  • ユーザー認証基盤が Cognito 前提: Cognito オーソライザーが第一選択。
  • 外部IdPや独自属性評価、複雑ポリシー: Lambda オーソライザー。
  • 実装速度・保守性重視: まず Cognito を検討し、要件不足時のみ Lambda 化。
よくある誤解
  • 「Lambda オーソライザーの方が常に高機能で正解」→ 過剰実装で運用負荷増になる場合がある。
  • 「Cognito なら細かい制御が全くできない」→ クレーム設計と API 側ポリシーで多くをカバー可能。
3分要点
  1. マネージドで足りるなら Cognito が試験的にも実務的にも優位。
  2. 独自認可ロジック必須なら Lambda オーソライザー。
  3. 「要件適合 + 運用負荷最小」が選定キーワード。
