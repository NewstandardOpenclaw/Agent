Topic 1: API Gateway Lambda オーソライザー（JWT/Cognito）設計の勘所

定義
  • Lambda オーソライザー は、API Gateway のリクエストを受けてトークン検証し、IAM ポリシーを返して許可/拒否を決める仕組み。
  • HTTP API + JWT オーソライザー は、Cognito や OIDC の JWT 検証をマネージドで実行できる方式。
重要性
  • DVA では「どこで認可判定するか」「トークン検証をどう効率化するか」が頻出。
  • アプリコードに認証ロジックを埋め込まず、API レイヤーで統制できると運用負荷が下がる。
使い分け
  • JWT の標準検証で足りる → HTTP API JWT オーソライザー（低遅延・低コスト）。
  • 独自ヘッダ・外部DB照合が必要 → Lambda オーソライザー。
  • API メソッドごとの差分制御が必要 → 返却ポリシーにリソース単位の許可/拒否を含める。
よくある誤解
  • 「オーソライザー結果キャッシュは常にONが正解」→ 権限変更の即時反映が要件なら TTL 設計が必要。
  • 「認証できたら全エンドポイント共通で許可」→ 最小権限の原則に反する。
3分要点
  1. 標準JWT検証ならマネージド方式を優先。
  2. 独自判定が必要なら Lambda オーソライザー。
  3. キャッシュTTLと最小権限が得点ポイント。

Topic 2: SQS + Lambda の部分バッチ失敗（Partial Batch Response）

定義
  • Partial Batch Response は、SQS トリガー Lambda でバッチ内の失敗メッセージだけ再処理対象にできる機能。
  • 成功分まで再実行される無駄を減らし、重複副作用リスクを抑える。
重要性
  • DVA のイベント駆動設計で「失敗時の再試行粒度」は高確率で問われる。
  • 大量メッセージ処理でコスト・遅延・重複処理の三重悪化を避けられる。
使い分け
  • 1件失敗で全件ロールバック不要 → Partial Batch Response を有効化。
  • 順序保証が絶対（FIFOで同一MessageGroupId）→ 失敗時の再処理戦略を慎重に設計。
  • 恒久失敗が混在する → DLQ と最大受信回数を併用して詰まりを防ぐ。
よくある誤解
  • 「バッチ内1件失敗は全部再試行が普通」→ 現在は失敗メッセージのみ返却できる。
  • 「DLQを付ければ再処理設計は不要」→ 再処理粒度・可観測性設計は別途必要。
3分要点
  1. 部分失敗返却で再処理を最小化。
  2. FIFOは順序要件を崩さない設計が前提。
  3. DLQは詰まり回避、冪等性は別途担保。
